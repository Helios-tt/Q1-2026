## Root cause analysis

- **Title**: Aztec RollupProcessor accepted rollups from non-provider callers while escape hatch was closed
- **Severity**: `critical`
- **Confidence**: `high`
- **Violated invariant**: A rollup state transition that can execute public withdrawals must only be callable by an authorized rollup provider unless the escape hatch is open.

### Final root cause

RollupProcessorV3.processRollup(bytes,bytes) documents that callers must be authorized rollup providers or the escape hatch must be open, but the function has only pause and reentrancy modifiers and performs no provider-or-escape-hatch check. In this transaction, attacker-routed helper contracts that were not rollup providers called processRollup while getEscapeHatchStatus() was false. The function then verified and applied rollup proof data, reached the proofId == 2 withdrawal branch, and transferred ETH/ERC20 assets from the Aztec proxy to the attacker.

### Recommended fixes

- At RollupProcessorV3.processRollup, require rollupProviders[msg.sender] or an open getEscapeHatchStatus() before decodeProof/processRollupProof can run.
- Add regression tests where non-provider contracts call processRollup while the escape hatch is closed and assert the call reverts before state, withdrawal, or fee effects.
