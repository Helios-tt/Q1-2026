## Root cause analysis

- **Title**: Bridge retry path released ERC20 vault funds using attacker-supplied message context
- **Severity**: `critical`
- **Confidence**: `medium`
- **Violated invariant**: A retried bridge message that releases canonical tokens from the L1 ERC20 vault must be authenticated as a previously proven source-chain ERC20 vault message with the same recipient, token, amount, and calldata.

### Final root cause

The bridge retry path invoked ERC20Vault.onMessageInvocation with context fields that matched the Taiko source ERC20 vault and with attacker-supplied token payload data. ERC20Vault trusted IBridge.context(), decoded the amount and recipient from the payload, and transferred canonical USDC from the vault. The exact MainnetBridge source branch and prior retry-state provenance are missing from supplied artifacts, so this is a partial RCA.

### Recommended fixes

- In MainnetBridge.retryMessage, bind retry execution to a proven/stored message commitment and reject any retry whose stored provenance does not exactly bind srcChainId, from, to, value, and data.
- Keep ERC20Vault.onMessageInvocation restricted to the resolved bridge, and treat bridge context as authoritative only after the bridge proves message provenance and replay/retry status.
