## Root cause analysis

- **Title**: OLPC pair-out transfer hook overburns AMM pool balance during public skim/sync reserve maintenance
- **Severity**: `critical`
- **Confidence**: `high`
- **Violated invariant**: A token transfer initiated by an AMM pair for skim/swap/liquidity maintenance must not debit more pool-owned token balance than the transfer amount authorized by the pair.

### Final root cause

OLPCToken._update burns value * decimalsValue whenever from == swapPair and the recipient is not tax-exempt, then sets the actual transfer value to zero. PancakePair.skim() can be called publicly to request transfer of only the pair's excess OLPC, but OLPC debits the pair by the multiplied amount; sync() then records the depleted balance as reserves. Repeating this sequence distorts the OLPC/LABUBU pair reserves and lets the downstream swap route extract USDT value.

### Recommended fixes

- In OLPCToken._update, forbid pair-originated burns from exceeding the requested transfer amount and avoid applying decimalsValue-multiplied burn logic to AMM pair maintenance transfers such as skim/sync/swap.
- Add invariant tests where PancakePair.skim() and sync() cannot reduce pair token balance by more than the actual pair excess selected by skim.
